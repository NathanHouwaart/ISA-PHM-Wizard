# Five Most Important Architectural Decisions

## AD-1: ISA-JSON Parsing Strategy

**Options**
- A) `isatools` as primary parser — objects come back as isatools domain objects
- B) Raw `json.load` + custom PHM-specific extraction
- C) `isatools` for structural validation, `json.load` for extraction into PHM classes

**Trade-offs**

| | isatools primary | json.load only | Hybrid (C) |
|---|---|---|---|
| Validation quality | ✅ Community-maintained | ❌ Hand-rolled | ✅ Both |
| Flexibility for PHM-specific fields | ❌ Fixed object model | ✅ Full control | ✅ Full control |
| Dependency risk | ⚠️ Heavy dep tree | ✅ Zero extra deps | ⚠️ isatools for validation only |
| LLM-tool serialization | ❌ Complex objects | ✅ Plain dicts | ✅ Plain dicts |

**Recommendation**: Option C — Hybrid. Use `isatools.isajson.load()` for validation against the ISA-JSON schema, then discard the isatools objects. Extract everything into PHM-specific Pydantic models using `json.load`. This decouples the extraction logic from isatools' object model, which is not designed for PHM's sensor-centric view.

---

## AD-2: Public API Shape — Fluent Chain vs Flat Functions

**Options**
- A) Fluent chain: `wrapper.study("case_01").assay("vibration").load_dataframe()`
- B) Flat: `wrapper.load_dataframe(study_id="case_01", assay_id="vibration")`
- C) Hybrid: both surfaces co-exist

**Trade-offs**

| | Fluent | Flat | Hybrid |
|---|---|---|---|
| Mirrors ISA hierarchy | ✅ | ❌ | ✅ |
| Supports progressive narrowing / discovery | ✅ | ❌ | ✅ |
| Implementation complexity | ⚠️ Proxy objects with state | ✅ Simple functions | ❌ Double the API |
| Agent tool compatibility | ❌ Poor — tools need flat calls | ✅ | ✅ |

**Recommendation**: Fluent API for interactive/notebook use, flat tool functions for the agent layer. The fluent proxies (`StudyProxy`, `AssayProxy`) are thin objects that hold a reference to the wrapper and a context tuple — they do not store loaded data themselves. The tool layer calls the flat methods directly on `ISAWrapper`.

---

## AD-3: DataFrame Loading Strategy — Eager vs Lazy

**Options**
- A) Eager: load all data files on `ISAWrapper.__init__()`
- B) Lazy per-call: load on each `load_dataframe()` call (no caching)
- C) Lazy with dict cache: load on first call, cache by `(study_id, assay_id, run_id)`

**Trade-offs**: Diagnostic datasets might have 10–30 assays. Prognostic datasets (e.g., FEMTO) can have 100+ runs × 3 sensors = 300+ files, each up to 100 MB. Eager loading is a non-starter. Pure lazy fails for repeated plots. LRU cache over-engineers the MVP.

**Recommendation**: Option C with a plain `dict` cache on the `DataIntegrator`. Key is `(study_id, assay_id, run_id)`. The user can call `integrator.clear_cache()` to free memory. Makes repeated calls in a notebook fast without complexity.

---

## AD-4: Run Layer — Always Explicit vs Transparent Default

**Options**
- A) Run is always explicit — even diagnostics require `run_id="run_01"`
- B) Run is invisible for diagnostic — `load_dataframe(study_id, assay_id)` only
- C) Run is optional — omit for diagnostics, required for prognostics, same API calls

**Trade-offs**: Option A creates friction for diagnostic users. Option B requires two separate code paths (diagnostic vs prognostic). Option C is the cleanest for users but needs careful API contracts to avoid ambiguity.

**Recommendation**: Option C. `run_id` is always `None` defaulted for diagnostic datasets (single run is selected automatically). For prognostic datasets, `run_id` is required for data-loading calls but optional for overview/listing calls. The `StudyProxy` exposes `has_runs: bool` so the user can branch correctly. Internally `run_id="run_01"` is always used.

---

## AD-5: Tool Layer Architecture — Module Functions vs ToolRegistry Class

**Options**
- A) Free module-level functions with global `wrapper` singleton
- B) `ToolRegistry` class that holds a `wrapper` reference and registers tools
- C) Provider-specific format directly (OpenAI `tools` list, Anthropic `tools` list)

**Trade-offs**: Global singletons are brittle. Provider-specific format creates vendor lock-in and complicates testing. `ToolRegistry` keeps tools testable (inject any wrapper instance), framework-agnostic (export to OpenAI or Anthropic format via adapter), and self-documenting (each tool has schema attached).

**Recommendation**: `ToolRegistry` with registered tool methods. Each tool is decorated with `@tool(name, description, input_schema)`. Export methods: `registry.to_openai_tools()`, `registry.to_anthropic_tools()`, `registry.to_dict()`. The agent calls `registry.call(tool_name, **kwargs)` which validates inputs, runs the tool, and returns a JSON-serializable dict.

# Five Biggest Scope Risks

## SR-1: ISA-JSON File Imperfection is Unbounded

**Why it's a risk**: Real datasets have broken relative paths, missing fields, non-standard ontology references, missing protocol links, and inconsistent naming conventions. The repair layer to handle all of these is effectively open-ended engineering work.

**What goes wrong in 12 weeks**: Student spends 4 of 12 weeks building an ISA-JSON repair tool, leaving only the core wrapper. The system only works on a single curated test file.

**Mitigation**: Define an "ISA-PHM valid subset" document (one A4 page). The `ISAPreprocessor` auto-fixes exactly these five issues only: (1) resolving relative paths to absolute given `data_root`, (2) normalizing file extensions to lowercase, (3) filling `name` if only `filename` is present, (4) stripping leading/trailing whitespace from `@id` values, (5) flagging assays with no linked data files as warnings. Everything else is a fatal `PreprocessingError` with a clear message. Refuse to continue on unknown structural issues.

---

## SR-2: Data File Linking Hides Deep Complexity

**Why it's a risk**: Files are referenced by relative path strings in the ISA-JSON. In practice, datasets are organized differently by every lab: flat directories, nested hierarchies, zip archives, HTTP URLs, Windows vs POSIX paths. Parsing a 2-column CSV sounds trivial — until you encounter semicolon separators, non-UTF-8 encoding, metadata header rows, mixed decimal separators, and variable-length header blocks.

**What goes wrong**: The student writes increasingly defensive CSV loading code, tests it on dataset A, and discovers dataset B breaks it. This can consume 3 weeks.

**Mitigation**: Declare a supported file contract in the MVP: CSV or TSV only, UTF-8 or Latin-1, comma or tab separator (auto-detected), exactly 2 columns, numeric values. Use `pandas.read_csv()` with `sep=None, engine="python"` for separator auto-detection. Skip the first N header rows when N is declared in protocol parameters. Log all loading decisions. If the file doesn't match, raise `DataFileError` immediately — no silent fallbacks.

---

## SR-3: Agent Scope Creep

**Why it's a risk**: Once a working LLM agent is demonstrated, stakeholders (supervisors, users) immediately request multi-step planning, code generation, report writing, and "just make it do X automatically." None of this is appropriate for a 12-week student project.

**What goes wrong**: The student shifts focus to the LLM interaction layer, builds increasingly complex prompts and workflows, and the wrapper itself is never properly tested or completed.

**Mitigation**: Hard constraint in the spec: the agent calls exactly one tool per turn, except for the patterns `list_studies → study_overview` and `list_assays → assay_overview` which are allowed as sequential pairs. The agent never calls more than 3 tools in a single response. The system prompt must include: "You are not an analyst. You are a dispatcher. Run one tool and summarize its output." Exclude code generation, report composition, and autonomous planning from phase 1–4. The agent is Phase 5.

---

## SR-4: Plotting Quality has No Natural Upper Bound

**Why it's a risk**: PHM practitioners expect domain-specific plots: envelope analysis, bearing frequency fault markers, waterfall diagrams, spectrogram overlays. Each plot type is a micro-project. Student will be tempted to extend the plotter continuously.

**What goes wrong**: 5 plot types become 12, none of them are properly documented, tested, or styled consistently. The plotter becomes the biggest and most fragile module.

**Mitigation**: Define MVP plots as exactly these six: (1) `plot_distribution` — histogram + KDE, (2) `plot_lifecycle` — time-series line plot, (3) `plot_correlation` — pairplot or correlation heatmap, (4) `plot_frequency_domain` — FFT magnitude spectrum, (5) `missing_values_report` — heatmap of NaN presence, (6) `plot_variability` — boxplot by run or condition. All plots must use a shared `PlotConfig` dataclass for title, xlabel, ylabel, figsize, colormap. No custom themes beyond this. Return `matplotlib.figure.Figure` always so the caller controls display/save.

---

## SR-5: Multi-Run Prognostic Complexity at Scale

**Why it's a risk**: Prognostic datasets like PRONOSTIA have 100+ runs per study, each with 3 sensor channels. That is 300+ files, each being a time-series snapshot. Listing runs, aggregating lifecycle metadata, and plotting degradation trends across all runs requires efficient batch loading, run indexing, and new plot types not needed for diagnostic.

**What goes wrong**: The wrapper works for diagnostic. When tested on PRONOSTIA-sized data, performance collapses. Run-level APIs were not designed for bulk access. The lifecycle plot function tries to load all 300 files sequentially.

**Mitigation**: Cap MVP at N≤50 runs per study. Implement `list_runs()` efficiently (pure metadata, no file loading). Design `plot_lifecycle()` to use a `run_ids` parameter that accepts a slice: `run_ids=range(0, 10)`. Document that full-dataset lifecycle plots require a future batch-loading layer. Test explicitly with the milling dataset (which has 17 runs — a safe scale for MVP).

# Full Technical Specification

## 1. Design Summary

The system is a Python wrapper first, LLM agent second. The wrapper implements all analytical intelligence: parsing, validation, metadata extraction, data loading, DataFrame integration, summarization, and plotting. The agent is a stateless dispatcher that maps natural language intents to pre-defined Python tool calls and summarizes their structured outputs. This inversion is non-negotiable: the LLM must never see raw ISA-JSON, must never compute numerical results independently, and must never invent descriptions of data it has not loaded.

The public API mirrors the ISA-PHM investigation hierarchy — `Investigation → Study → Assay → Run` — through a fluent proxy interface. This makes the library self-documenting and directly analogous to how PHM practitioners think about their experiments. A parallel flat tool API exposes the same capabilities to the agent layer through JSON-serializable function signatures.

## 2. Core Architectural Layers

### Layer 1: Parsing / Validation
**Responsibility**: Load ISA-JSON from file/string, validate structural correctness.
**Inputs**: File path or JSON string, optional `strict=True`.
**Outputs**: Validated raw Python dict (not isatools objects).
**Failure modes**: `ParseError` on malformed JSON; `ValidationError` on schema violations.
**Why it exists**: Separates format correctness from semantic correctness. isatools validates schema; our code extracts PHM semantics.

### Layer 2: Preprocessing / Repair
**Responsibility**: Apply safe, logged repairs to borderline-valid ISA-JSON before extraction.
**Inputs**: Raw dict from parser, `data_root: Path`.
**Outputs**: Repaired dict + `RepairLog` (list of applied fixes and warnings).
**Failure modes**: `PreprocessingError` on unfixable issues (missing assays, no studies).
**Why it exists**: Real datasets are imperfect. Separating repair from parsing means each can be tested in isolation and the user can inspect what was fixed.

### Layer 3: Metadata Extraction
**Responsibility**: Convert the raw dict into typed PHM domain objects (Pydantic models).
**Inputs**: Repaired dict.
**Outputs**: `InvestigationModel`, containing `StudyModel[]`, `AssayModel[]`, `RunModel[]`, `SensorModel[]`, etc.
**Failure modes**: `ExtractionError` on structurally valid but PHM-incompatible files.
**Why it exists**: Decouples the ISA schema from the PHM conceptual model. Downstream code works with typed objects, not dict navigation.

### Layer 4: Integration / Data Loading
**Responsibility**: Load raw and/or processed data files into pandas DataFrames, attach metadata columns.
**Inputs**: `AssayModel` or `RunModel`, `data_root`, cache dict.
**Outputs**: `pd.DataFrame` with columns `[time, value, study_id, assay_id, run_id, sensor_alias, unit]`.
**Failure modes**: `DataFileError` on missing file, encoding error, malformed CSV.
**Why it exists**: The analytical unit for PHM is a DataFrame with context. Merging metadata + measurements here avoids repetitive user code.

### Layer 5: Hierarchical Query / Navigation
**Responsibility**: Expose the Investigation → Study → Assay → Run tree through fluent proxy objects.
**Inputs**: `InvestigationModel`.
**Outputs**: `StudyProxy`, `AssayProxy`, `RunProxy` objects.
**Failure modes**: `KeyError`-style `StudyNotFoundError`, `AssayNotFoundError`.
**Why it exists**: Users and agents need to navigate the hierarchy interactively before committing to expensive data loading.

### Layer 6: Plotting / Analysis
**Responsibility**: Produce standard PHM-oriented plots and analysis results.
**Inputs**: `pd.DataFrame` or `AssayProxy`/`RunProxy`.
**Outputs**: `matplotlib.figure.Figure`.
**Failure modes**: `PlotError` on empty DataFrame, unsupported column type, missing time column.
**Why it exists**: PHM plots have domain conventions (FFT scaling, lifecycle x-axis). A shared plotter enforces consistency and removes boilerplate from user notebooks.

### Layer 7: Tool Layer (Agent Interface)
**Responsibility**: Translate tool calls into wrapper API invocations, serialize results to JSON.
**Inputs**: Tool name + JSON arguments (from LLM).
**Outputs**: JSON-serializable dicts (text, table as `list[dict]`, DataFrame summary, plot file path).
**Failure modes**: `ToolInputError` on invalid arguments; propagates wrapper errors as JSON error objects.
**Why it exists**: The agent cannot work with Python objects. This layer is the single translation boundary.

### Layer 8: UI / Notebook / CLI
**Responsibility**: Provide runnable notebooks with narrative, and a minimal CLI for batch use.
**Inputs**: Notebook cells, CLI commands.
**Outputs**: Rendered notebooks, exported CSV/PNG.
**Why it exists**: First delivery for end users. The Jupyter notebook is the primary UI in the MVP.

### Layer 9: Optional Web App
**Responsibility**: Serve the agent + wrapper as a REST API with a chat UI.
**Deferred**: Post-MVP. Not part of the 12-week scope.

`InvestigationModel`, `StudyModel`, `AssayModel`, `RunModel`, `SensorModel`, `ProtocolModel`, `FactorModel`, `FactorValueModel`, `DataFileModel`, `MetadataSummary`, `IntegratedDataset`

### What belongs to the agent tool layer (JSON-serializable dicts / Pydantic schemas)

## 6. Core Classes

### `ISAParser`
**Purpose**: Load and validate ISA-JSON.
**Constructor**: `ISAParser(strict: bool = True)`
**Key methods**:
- `load(path: Path | str) -> dict` — reads file, validates with isatools, returns raw dict
- `load_string(json_str: str) -> dict` — same from string

### `ISAPreprocessor`
**Purpose**: Apply safe repairs to the raw dict; produce a repair log.
**Constructor**: `ISAPreprocessor(data_root: Path, auto_fix: bool = True)`
**Key methods**:
- `preprocess(raw: dict) -> tuple[dict, RepairLog]`

**Internal repair actions** (exhaustive list for MVP):
1. Resolve relative `dataFiles[].name` paths against `data_root`
2. Normalize file extensions to lowercase
3. Fill `dataFiles[].name` from `filename` if `name` is empty
4. Strip whitespace from all `@id` values
5. Emit `WARNING` for assays with zero linked data files

### `MetadataExtractor`
**Purpose**: Convert the repaired raw dict into typed `InvestigationModel`.
**Constructor**: `MetadataExtractor()`
**Key methods**:
- `extract(repaired: dict) -> InvestigationModel`
- `_extract_study(study_dict: dict) -> StudyModel`
- `_resolve_sensor(assay_dict: dict, protocols: dict) -> SensorModel`
- `_build_factor_values(samples: list, factors: list) -> list[FactorValueModel]`

### `DataIntegrator`
**Purpose**: Load data files into DataFrames; attach metadata columns; cache results.
**Constructor**: `DataIntegrator(data_root: Path)`
**Key methods**:
- `load(assay: AssayModel, run_id: str, file_type: Literal["raw", "processed"] = "raw") -> pd.DataFrame`
- `clear_cache() -> None`
- `_read_file(path: Path) -> pd.DataFrame` — handles CSV/TSV, encoding detection
- `_attach_metadata(df, study_id, assay_id, run_id, sensor) -> pd.DataFrame`

**Cache key**: `(study_id, assay_id, run_id, file_type)` → `pd.DataFrame`

### `QueryNavigator`
**Purpose**: Entry point for fluent API; returns proxy objects.
**Constructor**: `QueryNavigator(investigation: InvestigationModel, integrator: DataIntegrator)`
**Key methods**:
- `study(study_id: str) -> StudyProxy`
- `list_studies() -> list[StudySummary]`

### `StudyProxy`
**Constructor**: `StudyProxy(study: StudyModel, integrator: DataIntegrator)`
**Key methods**:
- `overview() -> StudyOverview`
- `list_assays() -> list[AssaySummary]`
- `list_factors() -> list[FactorSummary]`
- `assay(assay_id: str) -> AssayProxy`
- `variable_overview() -> VariableOverview`
- `has_runs: bool` — property, True if `study.run_count > 1`

### `AssayProxy`
**Constructor**: `AssayProxy(assay: AssayModel, study: StudyModel, integrator: DataIntegrator)`
**Key methods**:
- `overview() -> AssayOverview`
- `list_runs() -> list[RunSummary]` — empty list if diagnostic

- `plot_lifecycle(time_column: str = "time") -> Figure`

- `plot_frequency_domain(column: str = "value", fs: float = None) -> Figure`

**Purpose**: Single public entry point. Facade over all layers.
**Constructor**: `ISAWrapper(path: Path, data_root: Path = None, auto_fix: bool = True)`

## 7. Public Python API

### Philosophy
The API mirrors ISA hierarchy. A user starts broad (`wrapper.investigation_overview()`), narrows to a study, then an assay, then optionally a run, then loads data or plots. Every step exposes what layers exist below it. No call should require the user to already know internal IDs — listing methods return both IDs and human-readable names.

### Naming Conventions
- `list_*()` — returns a list of summary objects, never loads data files
- `*_overview()` — returns a structured summary dict/object, never loads data files
- `load_dataframe()` — triggers file I/O, always explicit
- `plot_*()` — triggers file I/O (loads data if not cached), returns `Figure`
- `*_report()` — returns a structured analysis result object

### Return types
- Summary/overview methods → Pydantic model (serializable to dict via `.model_dump()`)
- Listing methods → `list[PydanticModel]`
- `load_dataframe()` → `pd.DataFrame`
- `plot_*()` → `matplotlib.figure.Figure`
- `*_report()` → domain-specific Pydantic model

### Full API Example Usage

```python
from isa_phm import ISAWrapper
from pathlib import Path

# Load and validate
wrapper = ISAWrapper(
    path=Path("dataset/isa_dataset.json"),
    data_root=Path("dataset/data/"),
    auto_fix=True
)

# Investigation level
overview = wrapper.investigation_overview()
print(overview.title, overview.experiment_type)

studies = wrapper.list_studies()
for s in studies:
    print(s.study_id, s.name, s.run_count)

# Study level
study = wrapper.study("case_01")
print(study.overview())
print(study.has_runs)            # False for diagnostic
print(study.list_assays())
print(study.list_factors())
print(study.variable_overview()) # factor names + values across runs

# Assay level
assay = study.assay("a_st01_se01")
print(assay.overview())          # sensor, measurement type, protocol params
print(assay.list_runs())         # [] for diagnostic

# Load data (diagnostic — run_id inferred automatically)
df = assay.load_dataframe()
print(df.head())                 # columns: time, value, study_id, assay_id, run_id, sensor_alias, unit

# Load data (prognostic — explicit run_id)
df = assay.load_dataframe(run_id="run_05")

# Plots
fig = assay.plot_distribution()
fig.savefig("dist.png")

fig = assay.plot_lifecycle()     # for prognostic: loads all cached runs

# Analysis
report = assay.missing_values_report()
print(report.n_missing, report.pct_missing)

outliers = assay.detect_outliers()
print(outliers.n_outliers, outliers.method)

# Run level (prognostic only)
run = assay.run("run_05")
print(run.overview())            # factor values for this run
df = run.load_dataframe()
fig = run.plot_lifecycle()

# Export
assay.export_clean_dataframe(Path("output/case01_vib_clean.csv"))
```

### Design Rules for Consistency
1. All ID parameters accept both the full `@id` string and the short name/alias interchangeably.
2. No method raises `AttributeError` — all return either a result or raise a typed `ISAPHMError` subclass.
3. Methods that return `Figure` never call `plt.show()` — caller controls display.
4. DataFrame columns are always the same set regardless of dataset; missing metadata columns are filled with `None`.
5. All summary/overview methods work without any data files being present.

## 8. Required Wrapper Capabilities

### Load and validate ISA-JSON
- Accept `Path` or string path
- Run isatools validation first; reject on schema violation
- Return typed domain model after successful extraction
- Log validation warnings (non-fatal issues) to `isa_phm` logger

### Optional preprocessing / repair
- Controlled by `auto_fix=True/False`
- When `auto_fix=False`, PreprocessingError is raised immediately on any fixable issue
- `RepairLog` always accessible via `wrapper.repair_log`

### Inspection of hierarchical structure
- `investigation_overview()` returns title, description, experiment_type, study count, contact count
- No file I/O required

### Listing studies / assays / runs
- `list_studies()` — `[{study_id, name, run_count, assay_count, factor_count}]`
- `list_assays(study_id)` — `[{assay_id, sensor_alias, measurement_type, run_count}]`
- `list_runs(study_id, assay_id)` — `[{run_id, run_number, factor_values: dict}]`

### Variable / sensor / parameter discoverability
- `variable_overview()` on `StudyProxy` returns factor matrix: rows=runs, columns=factors
- `assay.overview()` returns sensor metadata + protocol parameters inline
- All available column names in a DataFrame discoverable via `assay.columns()`

### Linking metadata to data files
- `DataIntegrator` resolves file paths relative to `data_root`
- Falls back with clear `DataFileError` if path resolution fails

### Producing integrated DataFrames
- Standard columns: `time, value, study_id, assay_id, run_id, sensor_alias, measurement_type, unit, file_type`
- Always `time` as float seconds if possible; keep original unit label in metadata

### Metadata summaries
- At all three levels (investigation / study / assay) via `*_overview()` methods

### Standard exploratory plots
- Six types defined — see Section 9

### Missing value analysis
- `missing_values_report()` on `AssayProxy` or `RunProxy`
- Returns: `{n_rows, n_missing, pct_missing, by_column: dict, plot: Figure}`

### Outlier analysis
- `detect_outliers(method="iqr")` — IQR method in MVP; `method="zscore"` as alternative
- Returns: `{n_outliers, pct_outliers, outlier_indices: list[int], threshold: float}`

### Export cleaned DataFrame
- `export_clean_dataframe(path: Path, drop_outliers: bool = False, fill_missing: str = None)`
- Supported `fill_missing`: `"ffill"`, `"bfill"`, `"interpolate"`, `None` (default: drop)

### Logging and transparent errors
- Module logger: `logging.getLogger("isa_phm")`
- Every repair action logged at `INFO`
- Every file load logged at `DEBUG`
- All errors include the offending `@id` and a suggested fix in the message

## 9. Plotting and Analysis API

All plotter methods live on `ISAPlotter` and are called by proxy objects. All return `matplotlib.figure.Figure`. All share a `PlotConfig(title, xlabel, ylabel, figsize, style)` with sensible defaults.

### `plot_distribution(df, column="value", bins=50)`
**Input**: `pd.DataFrame` with at least `[column]`, optional `hue` column
**Output**: `Figure` — histogram with KDE overlay
**Valid when**: DataFrame is non-empty, column is numeric
**Edge cases**: Raises `PlotError` if column is non-numeric or all-NaN; skips NaN rows silently with log warning

### `plot_lifecycle(df, time_column="time", value_column="value", group_by="run_id")`
**Input**: `pd.DataFrame` with time and value columns
**Output**: `Figure` — line plot, one line per `group_by` value if multiple groups
**Valid when**: `time_column` is numeric and monotonically increasing per group
**Edge cases**: Warns if time is non-monotonic; raises `PlotError` if fewer than 2 rows

### `plot_variability(df, value_column="value", group_by="run_id")`
**Input**: Grouped DataFrame
**Output**: `Figure` — boxplot per group
**Valid when**: At least 2 unique groups; column is numeric
**Edge cases**: `PlotError` if single group (use `plot_distribution` instead)

### `plot_correlation(df, columns=None)`
**Input**: `pd.DataFrame`, optional column list; if `None`, uses all numeric columns
**Output**: `Figure` — correlation heatmap (seaborn-style using matplotlib)
**Valid when**: At least 2 numeric columns
**Edge cases**: Drops non-numeric columns with warning; raises `PlotError` if fewer than 2 numeric columns remain

### `plot_frequency_domain(df, value_column="value", fs=None, window="hann")`
**Input**: `pd.DataFrame`, sampling frequency `fs` in Hz (required); Hann window applied by default
**Output**: `Figure` — FFT magnitude spectrum, x-axis in Hz, y-axis in dB
**Valid when**: Column is numeric, `fs > 0`, at least 64 samples
**Edge cases**: `PlotError` if `fs` is None and cannot be inferred; warns if signal length is not a power of 2 (zero-padding applied automatically)

### `plot_interactions(df, x, y, hue=None)`
**Input**: Column names for x, y, optional hue grouping
**Output**: `Figure` — scatter plot with optional color grouping
**Valid when**: All named columns exist and are numeric (hue can be categorical)
**Edge cases**: `PlotError` on missing columns; warns if fewer than 10 points

### `missing_values_report(df)`
**Input**: `pd.DataFrame`
**Output**: `MissingValuesReport` — includes counts, a `Figure` heatmap, per-column summary
**Always valid**: Returns zeroes and empty figure if no missing values

### `detect_outliers(df, columns=None, method="iqr", threshold=1.5)`
**Input**: `pd.DataFrame`, optional column list (defaults to all numeric), method, IQR multiplier
**Output**: `OutlierReport` — n_outliers, pct, outlier row indices, per-column results
**Valid when**: At least one numeric column
**Edge cases**: `PlotError` if no numeric columns; `method="zscore"` raises `ValueError` if threshold ≤ 0

### `export_clean_dataframe(df, path, drop_outliers=False, fill_missing=None)`
**Input**: `pd.DataFrame`, output `Path`, optional cleaning flags
**Output**: Writes CSV to path; returns path
**Valid when**: DataFrame is non-empty; parent directory exists or is created
**Edge cases**: `ExportError` on permission failure; warns if all rows dropped after filtering

## 10. Hierarchical Query / Navigation Design

### Discovery at each level

```
wrapper                               # ISAWrapper
  .investigation_overview()           # → InvestigationOverview (title, study count, experiment_type)
  .list_studies()                     # → list[StudySummary] (id, name, run_count, assay_count)
  .study("case_01")                   # → StudyProxy
    .overview()                       # → StudyOverview (factors, run_count, sensor list)
    .list_assays()                    # → list[AssaySummary] (id, sensor_alias, measurement_type)
    .list_factors()                   # → list[FactorSummary] (name, type, unit)
    .variable_overview()              # → VariableOverview (factor × run matrix as DataFrame)
    .has_runs                         # → bool
    .assay("a_st01_se01")             # → AssayProxy
      .overview()                     # → AssayOverview (sensor, protocols, parameter values)
      .columns()                      # → list[str] (available DataFrame columns)
      .list_runs()                    # → list[RunSummary] or [] for diagnostic
      .run("run_05")                  # → RunProxy (prognostic only)
        .overview()                   # → RunOverview (run_id, factor values)
        .factor_values()              # → dict[factor_name, value]
        .load_dataframe()             # → pd.DataFrame
```

### Diagnostic vs Prognostic behavior

| Operation | Diagnostic | Prognostic |
|---|---|---|
| `list_runs()` | Returns `[]` | Returns `[RunSummary × N]` |
| `load_dataframe()` | Infers `run_id="run_01"` | Requires `run_id` or raises `AmbiguousRunError` |
| `plot_lifecycle()` | Single line, time on x | One line per run, colored by run |
| `variable_overview()` | Single-row DataFrame | N-row DataFrame (one row per run) |
| `has_runs` | `False` | `True` |

### How sensors/parameters are discovered before plotting
```python
# Before plotting, user inspects what is available:
assay = wrapper.study("case_01").assay("a_st01_se01")
print(assay.overview())
# → AssayOverview(
#     assay_id="a_st01_se01",
#     sensor=SensorSummary(alias="vib_ch_1", type="Accelerometer", unit="g"),
#     measurement_type="Vibration",
#     protocol_params={"sampling_rate": 20000, "gain": 1.0, "filter": "anti-alias"},
#     data_files=[DataFileSummary(type="raw", path="...", exists=True)]
#   )

# Then discover available DataFrame columns:
print(assay.columns())
# → ["time", "value", "study_id", "assay_id", "run_id", "sensor_alias", "unit"]
```

## 11. Agent Tool Layer

All tools are registered on `ToolRegistry`. Each tool is a method decorated with `@tool(...)`. All return `dict` (JSON-serializable). The agent calls `registry.call(name, **kwargs)` which:
1. Validates input against the tool's Pydantic input schema
2. Runs the tool
3. Returns a JSON-serializable result dict
4. Catches all `ISAPHMError` subclasses and wraps them in `{"error": message, "code": error_code}`

### Tool Definitions

**`list_studies()`**
- Purpose: List all studies in the loaded dataset
- Arguments: none
- Returns: `{studies: [{study_id, name, run_count, assay_count}]}`
- Never does: loading data files

**`list_assays(study_id: str)`**
- Purpose: List all assays in a specific study
- Arguments: `study_id` (required)
- Returns: `{study_id, assays: [{assay_id, sensor_alias, measurement_type, run_count}]}`
- Validation: raises `ToolInputError` if `study_id` not found

**`list_runs(study_id: str, assay_id: str)`**
- Purpose: List all runs for a specific assay
- Returns: `{study_id, assay_id, runs: [{run_id, run_number, factor_values}]}`
- Returns `runs: []` for diagnostic datasets (not an error)

**`investigation_overview()`**
- Returns: `{title, description, experiment_type, study_count, contact_count, submission_date}`

**`study_overview(study_id: str)`**
- Returns: `{study_id, name, run_count, factors: [{name, type, unit}], assay_count, has_runs}`

**`assay_overview(study_id: str, assay_id: str)`**
- Returns: `{assay_id, sensor_alias, measurement_type, protocol_params: dict, data_files: [{type, path, exists}]}`

**`run_overview(study_id: str, assay_id: str, run_id: str)`**
- Returns: `{run_id, run_number, factor_values: dict, data_file: {type, path, exists}}`

**`variable_overview(study_id: str)`**
- Returns: `{study_id, factors: list, runs: list[{run_id, values: dict}]}`
- Equivalent to the factor-value matrix

**`load_dataframe(study_id: str, assay_id: str, run_id: str = None, file_type: str = "raw")`**
- Returns: `{study_id, assay_id, run_id, n_rows, n_cols, columns: list, head: list[dict], dtypes: dict, cache_hit: bool}`
- Never: returns the full DataFrame as JSON (too large)
- Always: loads and caches the real DataFrame internally for subsequent plot/analysis calls

**`plot_distribution(study_id, assay_id, run_id=None, column="value")`**
- Returns: `{plot_path: str, n_rows: int, column: str}` — saves PNG, returns path
- Requires: `load_dataframe` called first (or calls it internally)

**`plot_lifecycle(study_id, assay_id, run_ids=None)`**
- Returns: `{plot_path: str, n_runs: int}` for prognostic; `{plot_path, n_rows}` for diagnostic

**`plot_correlation(study_id, assay_id, run_id=None, columns=None)`**
- Returns: `{plot_path: str, n_columns: int, correlation_matrix: list[list[float]]}`
- Returns the correlation matrix as a nested list so the agent can describe it

**`missing_values_report(study_id, assay_id, run_id=None)`**
- Returns: `{n_rows, n_missing, pct_missing, by_column: dict, plot_path: str}`

**`detect_outliers(study_id, assay_id, run_id=None, columns=None, method="iqr")`**
- Returns: `{n_outliers, pct_outliers, method, by_column: dict}`

**`export_clean_dataframe(study_id, assay_id, run_id=None, path: str, drop_outliers=False, fill_missing=None)`**
- Returns: `{export_path: str, n_rows: int, n_dropped: int}`
- Validates: `path` must be under `data_root` or a designated output directory

### What tools must never do
- Access `raw_dict` directly — always use typed domain models
- Return full DataFrames as JSON
- Invent metadata not in the ISA file
- Make LLM calls
- Modify the ISA-JSON or any source data files

## 12. Agent Behavior Specification

The agent is a constrained tool-calling planner with these hard rules:

**Role**: The agent converts natural language questions into tool calls, executes them in sequence, and summarizes structured results in user-friendly language. It is not an analyst.

**Planning**: Before calling tools, the agent may call `list_studies()` or `investigation_overview()` if the study/assay identity is ambiguous.

**Ambiguity handling**: If the user says "show me the vibration data" and there are 3 assays with measurement_type="Vibration", the agent must ask "Which study are you interested in? Available studies are: [list]" before calling any data tool.

**Result summarization**: The agent must summarize tool outputs, not just repeat the raw JSON. E.g.: "Study case_01 has 3 assays measuring vibration, current, and acoustic emission. Run count is 1 (diagnostic)."

**Uncertainty**: If a tool returns an error, the agent reports it verbatim and asks the user how to proceed. It never retries silently.

**Constraints**:
- Max 3 tool calls per agent turn
- No code generation
- No autonomous multi-step planning beyond the immediate question
- No invented numerical values — every number cited must come from a tool result
- Claims about plots must reference the `plot_path` returned by the tool

## 14. Pydantic Schemas

```python
from pydantic import BaseModel, Field
from typing import Any, Literal, Optional
from pathlib import Path

# --- Domain model ---

class SensorSummary(BaseModel):
    alias: str
    technology_type: str
    measurement_type: str
    unit: Optional[str]

class AssaySummary(BaseModel):
    assay_id: str
    sensor: SensorSummary
    run_count: int

class StudySummary(BaseModel):
    study_id: str
    name: str
    run_count: int
    assay_count: int
    has_runs: bool

class RunSummary(BaseModel):
    run_id: str
    run_number: int
    factor_values: dict[str, Any]

class DataFileSummary(BaseModel):
    type: Literal["raw", "processed"]
    path: str
    exists: bool

# --- Overview objects ---

class InvestigationOverview(BaseModel):
    title: str
    description: str
    experiment_type: Literal["diagnostic", "prognostic"]
    study_count: int
    contact_count: int

class StudyOverview(BaseModel):
    study_id: str
    name: str
    run_count: int
    has_runs: bool
    factors: list[dict]
    assay_count: int

class AssayOverview(BaseModel):
    assay_id: str
    sensor: SensorSummary
    measurement_type: str
    protocol_params: dict[str, Any]
    data_files: list[DataFileSummary]

# --- Tool I/O ---

class DataFrameRef(BaseModel):
    study_id: str
    assay_id: str
    run_id: Optional[str]
    n_rows: int
    n_cols: int
    columns: list[str]
    head: list[dict]        # first 5 rows as list of dicts
    cache_hit: bool

class PlotRef(BaseModel):
    plot_path: str
    plot_type: str
    description: str

class MissingValuesReport(BaseModel):
    n_rows: int
    n_missing: int
    pct_missing: float
    by_column: dict[str, int]
    plot_path: Optional[str]

class OutlierReport(BaseModel):
    n_outliers: int
    pct_outliers: float
    method: str
    by_column: dict[str, dict]

class ExportResult(BaseModel):
    export_path: str
    n_rows: int
    n_dropped: int

class ToolError(BaseModel):
    error: str
    code: str    # e.g., "STUDY_NOT_FOUND", "DATA_FILE_MISSING"
    suggestion: Optional[str]
```

## 15. Error Handling and Transparency

### Custom exception hierarchy

```python
class ISAPHMError(Exception): ...

class ParseError(ISAPHMError): ...          # malformed JSON
class ValidationError(ISAPHMError): ...     # ISA schema violation
class PreprocessingError(ISAPHMError): ...  # unfixable ISA-JSON issue
class ExtractionError(ISAPHMError): ...     # valid ISA but not PHM-compatible
class DataFileError(ISAPHMError): ...       # file not found, encoding error
class StudyNotFoundError(ISAPHMError): ...  # study_id not in dataset
class AssayNotFoundError(ISAPHMError): ...  # assay_id not in study
class RunNotFoundError(ISAPHMError): ...    # run_id not in assay
class AmbiguousRunError(ISAPHMError): ...   # prognostic assay, no run_id given
class PlotError(ISAPHMError): ...           # cannot produce plot (bad data)
class ExportError(ISAPHMError): ...         # cannot write output file
class ToolInputError(ISAPHMError): ...      # agent tool received invalid args

```

### Error message contract
Every `ISAPHMError` message must include:
- The offending entity ID (study_id, assay_id, path, etc.)
- The expected format or valid options
- A suggested fix

Example: `StudyNotFoundError: Study ID 'case_99' not found. Available study IDs: ['case_01', 'case_02', 'case_03']`

### User-facing vs developer-facing
- User-facing: `ISAPHMError` subclasses — clear English, no tracebacks unless `debug=True`
- Developer-facing: Full Python tracebacks via standard logging at `DEBUG` level
- Tool layer: All `ISAPHMError` subclasses are caught and returned as `ToolError` JSON — never propagate raw Python exceptions to the LLM

## 16. Preprocessing / Repair Strategy

### What can be auto-fixed (with INFO log)
1. Relative `dataFiles[].name` paths → resolved against `data_root`
2. Uppercase file extensions → lowercased (`.CSV` → `.csv`)
3. Missing `name` field when `filename` field is present → copy
4. Leading/trailing whitespace in `@id` strings → stripped
5. Empty `description` fields → filled with `""` (not `None`)

### What triggers WARNING but continues
- Assays with no linked data files present (metadata-only operation will still work)
- Protocol parameters with unrecognized unit annotations
- Factor values referencing unknown factor IDs (warning + skip that value)
- Duplicate `@id` values (first occurrence wins, all others logged)

### What is FATAL (stops processing immediately)
- Zero studies in the investigation
- Study with no assays
- `@id` references that cannot be resolved after repair (circular or missing)
- Data file paths that resolve outside the allowed `data_root` directory (security: path traversal)
- JSON that does not validate against the ISA-JSON schema at all

### Repair log format

```python
class RepairAction(BaseModel):
    severity: Literal["INFO", "WARNING"]
    entity_type: str        # "DataFile", "Factor", "Protocol", etc.
    entity_id: str
    field: str
    old_value: Any
    new_value: Any
    message: str

class RepairLog(BaseModel):
    actions: list[RepairAction]
    fatal_error: Optional[str]

    def summary(self) -> str: ...   # human-readable one-liner per action
```

```python
# isa_phm/wrapper.py

from pathlib import Path
from .parser import ISAParser
from .preprocessor import ISAPreprocessor
from .extractor import MetadataExtractor
from .integrator import DataIntegrator
from .navigator import QueryNavigator
from .schemas import InvestigationOverview, RepairLog

class ISAWrapper:
    """
    Main entry point for the ISA-PHM Python wrapper.
    Facade over all internal layers.
    """

    def __init__(
        self,
        path: Path | str,
        data_root: Path | str = None,
        auto_fix: bool = True,
        strict_validation: bool = True,
    ):
        path = Path(path)
        data_root = Path(data_root) if data_root else path.parent

        parser = ISAParser(strict=strict_validation)
        raw = parser.load(path)

        preprocessor = ISAPreprocessor(data_root=data_root, auto_fix=auto_fix)
        repaired, self.repair_log = preprocessor.preprocess(raw)

        extractor = MetadataExtractor()
        self._investigation = extractor.extract(repaired)

        self._integrator = DataIntegrator(data_root=data_root)
        self._navigator = QueryNavigator(self._investigation, self._integrator)

    def investigation_overview(self) -> InvestigationOverview:
        return self._navigator.investigation_overview()

    def list_studies(self) -> list:
        return self._navigator.list_studies()

    def study(self, study_id: str):
        return self._navigator.study(study_id)

    @property
    def repair_log(self) -> RepairLog:
        return self._repair_log
```

```python
# isa_phm/navigator.py

from .schemas import StudyProxy, AssayProxy, RunProxy, InvestigationOverview, StudySummary
from .errors import StudyNotFoundError

class QueryNavigator:
    def __init__(self, investigation, integrator):
        self._inv = investigation
        self._integrator = integrator
        self._study_index = {s.study_id: s for s in investigation.studies}

    def investigation_overview(self) -> InvestigationOverview:
        return InvestigationOverview(
            title=self._inv.title,
            description=self._inv.description,
            experiment_type=self._inv.experiment_type,
            study_count=len(self._inv.studies),
            contact_count=len(self._inv.contacts),
        )

    def list_studies(self) -> list[StudySummary]:
        return [
            StudySummary(
                study_id=s.study_id,
                name=s.name,
                run_count=s.run_count,
                assay_count=len(s.assays),
                has_runs=s.run_count > 1,
            )
            for s in self._inv.studies
        ]

    def study(self, study_id: str) -> "StudyProxy":
        study = self._study_index.get(study_id)
        if study is None:
            available = list(self._study_index.keys())
            raise StudyNotFoundError(
                f"Study ID '{study_id}' not found. Available: {available}"
            )
        return StudyProxy(study=study, integrator=self._integrator)
```

```python
# isa_phm/tools.py

import functools
from typing import Callable
from .wrapper import ISAWrapper
from .schemas import ToolError

def tool(name: str, description: str, input_schema: dict):
    """Decorator to register a method as an agent tool."""
    def decorator(fn: Callable):
        fn._tool_name = name
        fn._tool_description = description
        fn._tool_schema = input_schema
        return fn
    return decorator

class ToolRegistry:
    def __init__(self, wrapper: ISAWrapper):
        self._wrapper = wrapper
        self._tools = {
            name: method
            for name, method in self.__class__.__dict__.items()
            if hasattr(method, "_tool_name")
        }

    def call(self, tool_name: str, **kwargs) -> dict:
        fn = self._tools.get(tool_name)
        if fn is None:
            return ToolError(error=f"Unknown tool: {tool_name}", code="TOOL_NOT_FOUND").model_dump()
        try:
            return fn(self, **kwargs)
        except Exception as e:
            return ToolError(error=str(e), code=type(e).__name__).model_dump()

    @tool("list_studies", "List all studies in the loaded dataset", {})
    def list_studies(self) -> dict:
        studies = self._wrapper.list_studies()
        return {"studies": [s.model_dump() for s in studies]}

    @tool(
        "load_dataframe",
        "Load a data file into memory for a specific study/assay/run",
        {
            "study_id": {"type": "string", "required": True},
            "assay_id": {"type": "string", "required": True},
            "run_id": {"type": "string", "required": False},
            "file_type": {"type": "string", "enum": ["raw", "processed"], "default": "raw"},
        }
    )
    def load_dataframe(self, study_id: str, assay_id: str, run_id: str = None, file_type: str = "raw") -> dict:
        assay_proxy = self._wrapper.study(study_id).assay(assay_id)
        df = assay_proxy.load_dataframe(run_id=run_id, file_type=file_type)
        return {
            "study_id": study_id,
            "assay_id": assay_id,
            "run_id": run_id,
            "n_rows": len(df),
            "n_cols": len(df.columns),
            "columns": list(df.columns),
            "head": df.head(5).to_dict(orient="records"),
        }

    def to_openai_tools(self) -> list[dict]:
        return [
            {
                "type": "function",
                "function": {
                    "name": fn._tool_name,
                    "description": fn._tool_description,
                    "parameters": {"type": "object", "properties": fn._tool_schema},
                }
            }
            for fn in self._tools.values()
        ]
```

## 22. Final Recommendation

**Architecture**: Layered Python package with strict one-way dependency flow: `ISAWrapper` → `QueryNavigator` → `DataIntegrator` → `ISAPlotter`. The `ToolRegistry` sits alongside `ISAWrapper` and wraps it — it does not own any layer directly. The LLM agent is a side car, never a layer in the main chain.

**Public API style**: Fluent chain for interactive use, flat dict-returning functions for tool use. Do not conflate them — keep proxy objects in `navigator.py` and tool functions in `tools.py`. Both call the same underlying `DataIntegrator` and `ISAPlotter` methods.

**Tool API style**: JSON-in, JSON-out, no Python objects crossing the tool boundary. Every tool returns either a valid result dict or a `ToolError` dict — never raises an exception to the agent.

**Cut if time is short** (in order): `RunProxy` (use `AssayProxy.load_dataframe(run_id=...)` directly) → `detect_outliers` → `plot_correlation` → `export_clean_dataframe` → agent runner (deliver ToolRegistry + notebook demo instead).

**Must be implemented correctly first**: `MetadataExtractor` producing accurate `InvestigationModel` and `DataIntegrator` linking file paths to DataFrames. Everything else — plotting, agent, web — depends on these two working reliably. Do not start Phase 3 before Phase 1 is fully tested.
